# Известные проблемы и решения

**Последнее обновление:** 2026-05-04

Файл содержит только актуальные риски и спорные архитектурные решения. Устранённые проблемы перенесены в раздел «Исправлено».

Активные команды AutoCAD: `MERGEDWG`, `MERGEDWG_DIAG_TEST`, `SMART_MERGE_TEXT`, `CreateETransmitZip`, `MergeTextStyles`, `JOIN_LINES`.

---

## Открытые проблемы

### KI-1. `DuplicateRecordCloning.Ignore` может унаследовать целевые стили

**Где:** `BlockInserter.InsertNativeObjects`

`WblockCloneObjects` с `DuplicateRecordCloning.Ignore` избегает дублирования стилей и слоёв в целевой базе. Но если исходный стиль совпадает по имени с целевым, а их параметры различаются — клонированные объекты унаследуют целевое определение.

**Решение:** Оставить текущее поведение (стабильность целевой базы) или перейти на `MangleName` при необходимости точного воспроизведения внешнего вида.

---

### KI-2. Нет отмены для длинных операций

**Где:** `CombineCommands`, `CombineOrchestrator`, `LayoutProjectionProcessor`

При большом количестве файлов операция может выполняться долго без возможности отмены.

**Решение:** Добавить `CancellationToken` через весь пайплайн с проверкой между файлами и перед тяжёлыми операциями листа.

---

### KI-3. Жёстко заданные операционные ограничения

| Ограничение | Значение | Место |
|---|---|---|
| Максимальный размер файла | 15 МБ | `FileUtil.GetFiles` |
| Глубина рекурсии в папках | 3 | `FileUtil.GetFiles` |
| Ограничение масштаба VP | 1:100 | `LayoutProjectionProcessor` |

Значения консервативны и не настраиваются пользователем.

**Решение:** Добавить конфигурационный файл только если реальные проекты потребуют других порогов.

---

### KI-4. Визуальное качество размеров требует ручной проверки

**Серьёзность:** Средняя

**Где:** `DimensionStyleNormalizer`, `LayoutProjectionProcessor`

Алгоритм нормализует размеры в подготовленной исходной базе до `WblockCloneObjects`. Для каждого размера создаётся стиль вида `"{Стиль} - Scale {Множитель}"`, очищаются DSTYLE-overrides, `Dimscale` и `Dimlfac` приводятся к `1.0`.

**Риск:** После масштабирования Model Space (clamp), переноса Paper Space или трансформации aux-VP высота текста, размер стрелок или числовое значение размера может не совпасть с оригиналом.

**Статус:** Диагностика (`[DIM-NORMALIZE]`, `[DIM-STYLE]`) пишется в лог. Требуется ручная визуальная QA на репрезентативных листах.

## Исправлено

| Проблема | Исправление |
|---|---|
| Дублирование XData-парсинга (80 строк) | Удалены из `DimensionStyleDiagnosticUtils`; остался только в `DimensionUtils`. |
| Мёртвые методы `LogNewStylesBeforeMerge`, `ClearDimensionOverrides` | Удалены из `DimensionStyleDiagnosticUtils`. |
| Класс-обёртка `FolderSelector` | Удалён; `CombineCommands` вызывает `UiDialogService` напрямую. |
| Дублирование в `AcadWarningSuppressScope` / `AcadUnitScalingOverrideScope` | Общая логика вынесена в базовый `SysVarScope`. |
| Двойные транзакции в `LayoutUtil.GetPaperSpaceEntities` | Инлайнена логика поиска layout; одна транзакция вместо двух. |
| Дрейф масштаба aux-VP | Трансформация aux-to-main использует исходный масштаб main VP до глобального clamp. |
| Удвоенная нормализация размеров | Удалён пост-merge healer; нормализация — один раз в `DimensionStyleNormalizer` до клонирования. |
| Авто-масштаб ×304.8 на имперских DWG | Синхронизация `Insunits`/`Measurement` перенесена после `CloseInput(true)`. |
| Остатки оригиналов aux-VP | Добавлен `EraseEntitiesOutsideMainWindow` после клонирования aux. |
| `ProgressMeter` не закрывался при ошибке | `pm.Stop()` перенесён в `finally`. |
