# Алгоритм объединения DWG-файлов

**Метаданные документа**
- Актуализировано: 2026-04-23
- Источник истины: `Application/Commands/MergeCommands.cs`, `Application/Merge/*`, `Application/Merge/Layouts/*`, `Application/Utils/*`

Этот документ описывает фактический пошаговый процесс работы AutoBIMFusion при выполнении команды `MERGEDWG`.

## Общий процесс

1. Пользователь выбирает папку с DWG.
2. Файлы перечисляются рекурсивно (с ограничениями по глубине, префиксу и размеру).
3. Каждый DWG валидируется и экспортируется во временный DWG из первого листа (Paper Space -> Model Space).
4. Объекты из временного DWG вставляются в целевой чертёж как **нативные сущности** (без Xref/Bind), со смещением по оси X.
5. Пути растровых изображений нормализуются и изображения копируются рядом с целевым DWG.
6. Результат сохраняется в `AC1032`, выполняются `REGENALL` и `ZOOM EXTENTS`.

---

## Подробные шаги алгоритма

### Шаг 1: Запуск команды и подготовка

- Пользователь запускает `MERGEDWG`.
- Повторный параллельный запуск блокируется через `SemaphoreSlim _mergeGate`.
- Через `FolderSelector` выбирается исходная папка.
- `FileEnumerator.GetFiles(...)` собирает `*.dwg`:
  - рекурсия в подпапки: `MaxRecursionDepth = 3`,
  - исключаются файлы с префиксом `#`,
  - исключаются файлы больше `15 МБ`,
  - итог сортируется natural-sort сравнителем `WindowsNaturalComparer`.

### Шаг 2: Блокировка документа и расчёт пути сохранения

- Все операции merge выполняются внутри `doc.LockDocument()`.
- Путь результата строится методом `BuildSavePath(rootPath)`:
  - если выбрана папка `X\Folder`, результат: `X\Folder.dwg` (в родительской директории выбранной папки).

### Шаг 3: Обработка каждого файла (`DwgMerger.MergeSingleFile`)

Для каждого файла:

1. Валидация:
   - `FileHelper.TryValidateFile(..., FileShare.ReadWrite, ...)`,
   - `FileHelper.TryValidateDwgStructure(...)`.
2. Экспорт первого листа во временный DWG через `ViewportLayoutExporter.ExportToTempAsync(...)`.
3. Чтение границ временного файла (`ReadBounds` через `Database.Extmin/Extmax`).
4. Вставка в целевую БД методом `BlockInserter.InsertNativeObjects(...)`.
5. Удаление временного файла в `finally`.

Если на любом этапе файл не подходит/пустой/не вставлен, возвращается `MergeResult.Warn` или `MergeResult.Fail`.

### Шаг 4: Экспорт листа во временный DWG (`ViewportLayoutExporter`)

- Открывается исходный документ DWG.
- Находится первый лист (`LayoutUtil.TryFindFirstLayout`).
- Работа выполняется в `ExecuteInCommandContextAsync`.
- В зависимости от числа viewport на листе:
  - `0 VP`: Paper Space переносится в Model Space с масштабом `1:100` (`MaxScaleMultiplier`).
  - `1 VP`: перенос через матрицу `BuildPaperToMainMatrix(...)`.
  - `2+ VP`: выбирается главный VP, вспомогательные окна дополнительно переносят модельные объекты матричными преобразованиями.
- Выполняется `ModelSpaceTrimmer.TrimOutside(...)` по вычисленным границам рамки.
- Нормализуются пути `RasterImageDef` (`FindFile` + `Load`).
- Устанавливается `TILEMODE = 1`, выполняется `REGEN`.
- Временный DWG сохраняется как `AC1032`.

### Шаг 5: Вставка в целевой DWG (`BlockInserter`)

- Временный DWG открывается на чтение.
- Все сущности его Model Space клонируются в Model Space целевой БД через `WblockCloneObjects`.
- Клонированные сущности смещаются матрицей `Displacement`.
- Позиция рассчитывается по правой границе предыдущей вставки (`_rightMax`) и зазору `gapPercent`.
- Возвращаются мировые границы вставленных объектов.

> В текущей реализации **не используется** attach/bind Xref. Вставка выполняется как перенос нативной геометрии.

### Шаг 6: Пост-обработка, сохранение и завершение

После обработки всех файлов:

1. `RasterImagePathFixer.CopyImagesToTargetFolder(...)`:
   - копирует найденные изображения в папку целевого DWG,
   - обновляет `RasterImageDef.SourceFileName` на относительное имя файла,
   - устраняет коллизии имён (`name_1.ext`, `name_2.ext`, ...).
2. `SaveMerged(...)`:
   - удаляет существующий целевой файл,
   - сохраняет БД через `db.SaveAs(savePath, DwgVersion.AC1032)` в suppress-scope.
3. Выполняются `REGENALL` и `ZOOM EXTENTS`.
4. Показывается сводка по `MergeStatistics`.

## Ключевые свойства текущего алгоритма

- Обрабатывается первый найденный layout каждого DWG.
- Временные файлы удаляются в `finally`.
- Все merge-операции выполняются под document lock.
- Ошибки и предупреждения логируются через `OperationLogger` (Editor + файл Serilog).
- Итоговая геометрия размещается последовательно по оси X без взаимных пересечений (через расчёт `_rightMax` + `gap`).
